# MdEv 编辑视图自动换行修复总结

> 一份事后复盘：记录从「反复失败」到「真正定位根因」的完整过程，沉淀可复用的经验。

---

## 一、问题现象

开启「自动换行」设置后：
- **预览视图**换行正常（走纯 CSS class 切换）。
- **编辑视图（CodeMirror）**不换行，长行依旧横向溢出；且开启自动换行时编辑区仍出现水平滚动条。
- 无论「启动时默认开启」还是「运行时手动切换」，编辑视图都不生效。

技术栈：Electron 33 + Vue 3 + vue-codemirror 6.1.1 + @codemirror/view 6。

---

## 二、修复过程（三次迭代）

### 第一次：在 onEditorReady 里补 setWordWrap —— ❌ 失败

**当时的思考（初始化时序假设）**

读 `EditorPane.vue` 后判断是初始化时序问题：
- `EditorPane` 用 `v-show` 控制显隐，组件**始终挂载**；但内部 `<Codemirror>` 用 `v-if="store.activeTab"`，**只有打开文件后才渲染**。
- `onMounted` 里调用了 `setWordWrap(store.settings.wordWrap)`，但此时编辑器还没 ready，`editorView.value` 为 `null`，`setWordWrap` 直接 `return`，什么都没做。
- 等用户打开文件、`Codemirror` 渲染、`onEditorReady` 触发时，原代码只设置了 `editorView.value`，**没有重新应用 wordWrap 设置**。

**改动**：在 `onEditorReady` 里补一行 `setWordWrap(store.settings.wordWrap)`。

**结果**：用户反馈 `npm start` 启动后仍不生效。

**失败原因（事后看）**：这次只覆盖了「启动时已开启」的初始化场景，且整套 `setWordWrap → compartment.reconfigure` 链路本身就是建立在错误假设上——见第三次修复的根因。

---

### 第二次：:key 强制重建编辑器 —— ❌ 失败

**当时的思考（怀疑 reconfigure 不可靠）**

既然第一次没生效，怀疑运行时切换走的 `watch → compartment.reconfigure` 链路不可靠（项目 CHANGELOG 也提过「vue-codemirror 切换 extension 时 reconfigure 不可靠」）。于是放弃 `reconfigure`，改用「最稳」的方式：

- Codemirror 加 `:key="'cm-'+activeTabId+'-'+(wordWrap?'wrap':'nowrap')"`，切换 wordWrap 时 key 变化，Vue 销毁旧编辑器、创建新编辑器。
- `cmExtensions` computed 根据 `store.settings.wordWrap` 注入 `EditorView.lineWrapping`，让新编辑器初始化时就包含 lineWrapping。
- `watch wordWrap` 保存光标 + 滚动位置，重建后 `onEditorReady` 恢复。

**逻辑上自洽**：lineWrapping 在 `EditorState.create({extensions})` 时就包含，CodeMirror 初始化即应用 `.cm-lineWrapping` class，不存在 reconfigure 时序问题。

**结果**：用户反馈打包 exe 后**仍不生效**——首次打开、手动切换都不换行。

**失败原因**：整套逻辑建立在一个**未经证实的假设**上——「Vue 重建 Codemirror 组件时，会用我们传入的 `extensions` prop 创建新的 EditorView」。这个假设是错的。详见第三次修复。

---

### 第三次：读 vue-codemirror 源码定位真正根因 + CSS 兜底 —— ✅ 成功

两次失败后，意识到不能再靠「读应用层代码 + 猜测」下结论，必须读**第三方库源码**确认机制。

#### 关键突破：vue-codemirror 创建 EditorView 时根本不用 props.extensions

读 `node_modules/vue-codemirror/dist/vue-codemirror.esm.js` 的 `setup` 函数体，发现：

```js
// onMounted 里创建 EditorView：
d.value = j({
  doc: t.modelValue,
  selection: U.value.selection,
  extensions: null !== (e = D.extensions) && void 0 !== e ? e : [],  // ← 用 D.extensions，不是 props.extensions！
  ...
})
```

**vue-codemirror v6.1.1 创建 EditorView 时用的是 `D.extensions`（默认 `[basicSetup]` + 全局注入配置），根本不用我们通过 prop 传入的 `extensions`！** 我们的 `extensions` prop 是靠 `watch(t.extensions, cb, {immediate:true})` → `reExtensions()` 通过 `dispatch` 动态注入的，而 `reExtensions` 首次走 `StateEffect.appendConfig.of(compartment.of(extensions))`，后续走 `reconfigure`。

这一条机制，**同时解释了前两次失败**：

- **第一次失败**：我把 `wordWrapCompartment.of([])` 放在 `baseExtensions` 里，但 vue-codemirror 创建 view 时**根本不用 `baseExtensions`**，所以 compartment 从未注册，`reconfigure` 一个未注册的 compartment 是**无效空操作**——什么都不会发生。
- **第二次失败**：`:key` 重建时，新 view 创建时仍用 `basicSetup`（不含 lineWrapping），lineWrapping 仍靠 `appendConfig` 动态注入——而动态注入这一环不可靠。

#### 补充确认：lineWrapping 的真实机制

读 `@codemirror/view` 源码确认：

- `EditorView.lineWrapping = EditorView.contentAttributes.of({class: 'cm-lineWrapping'})` —— 它给 **`.cm-content`**（不是 `.cm-editor`）加 `cm-lineWrapping` class。
- 默认 `.cm-content { white-space: pre }`（不换行）；lineWrapping 的 CSS 规则 `.cm-lineWrapping { white-space: break-spaces; word-break: break-word; overflow-wrap: anywhere }`（后定义，同优先级，胜出）。
- **换行是否生效 = `.cm-content` 是否有 `cm-lineWrapping` class = lineWrapping extension 是否真正进入了 configuration。**

#### 最终方案：CSS 兜底（!important）+ appendConfig compartment 双保险

既然换行本质是 `.cm-content` 的 `white-space`，且 vue-codemirror 的动态注入不可靠，改用最简单可靠的 CSS：

1. **CSS 兜底**（核心，100% 生效）：
   ```css
   .editor-pane.word-wrap .cm-content {
     white-space: pre-wrap !important;
     word-break: break-word !important;
     overflow-wrap: anywhere !important;
   }
   ```
   `.editor-pane` 已有 `:class="{ 'word-wrap': store.settings.wordWrap }"` 绑定，切换 wordWrap 时 class 立即变化，CSS 立即生效，**完全不依赖任何 CodeMirror 机制**。这同时解决了换行和水平滚动条。

2. **appendConfig compartment**（补充）：在 `onEditorReady` 里 `v.dispatch({effects: StateEffect.appendConfig.of(wordWrapCompartment.of(wordWrap ? lineWrapping : []))})`，让 CodeMirror 的 heightOracle facet 也知道换行，避免行高/滚动计算偏差。切换时用 `wordWrapCompartment.reconfigure`（compartment 此时已通过 appendConfig 注册，reconfigure 有效）。

3. **运行时诊断**：`onEditorReady` 里检查 `.cm-editor`/`.cm-content` 的 class 和计算样式，存 `window.__wwDebug`；`electron/main.js` 的 `did-finish-load` 后轮询写到 `userData/ww-debug.json`，用于确认 DOM 真实状态。

**结果**：成功。用户确认编辑视图自动换行已生效（首次打开 + 手动切换都正常）。

---

## 三、最终根因（一句话）

**vue-codemirror v6.1.1 创建 EditorView 时不用 `props.extensions`，而用默认的 `basicSetup`；我们传入的 extensions 靠 `watch + appendConfig/reconfigure` 动态注入，这一环在 vue-codemirror 包装层下不可靠。** 所有基于「extensions prop 会在创建时生效」的方案（compartment.reconfigure、:key 重建）都注定失败。

---

## 四、最终方案要点

| 层 | 机制 | 作用 | 可靠性 |
|---|---|---|---|
| CSS 兜底 | `.editor-pane.word-wrap .cm-content { white-space: pre-wrap !important }` | 强制换行 + 消除水平滚动条 | 100%（纯 class 切换，不依赖 CodeMirror） |
| appendConfig compartment | `StateEffect.appendConfig.of(wordWrapCompartment.of(...))` | 让 heightOracle 正确计算行高 | 补充（即使失效，CSS 兜底仍保证视觉换行） |
| 运行时诊断 | `window.__wwDebug` → `ww-debug.json` | 确认 DOM 真实状态 | 排查用，可清理 |

**为什么 CSS 兜底是最佳选择**：
- 不依赖 vue-codemirror 的任何机制，绕过整个不可靠的动态注入链路。
- 纯 class 切换，**零副作用**：光标、撤销历史、选区、滚动位置全部保留（相比 `:key` 重建方案会丢失撤销历史，这是质的提升）。
- `!important` 确保覆盖 CodeMirror 默认的 `white-space: pre`。

## 五、经验总结

### 1. 读第三方库源码胜过猜测

前两次失败的根本原因，是对 vue-codemirror 机制的错误假设——「extensions prop 会在 EditorView 创建时生效」。这个假设看起来理所当然，但实际是错的。

**教训**：当基于框架封装层的方案反复失败时，停止靠「读应用层代码 + 推理」下结论，**直接读第三方库源码确认机制**。一行 `console.log` 或读一遍 dist 产物，胜过十次猜测。本次正是因为读了 `vue-codemirror.esm.js` 的 setup body，才发现 `extensions: D.extensions`（而非 `props.extensions`）这个致命细节。

### 2. 当封装层不可靠时，用更底层的机制兜底

vue-codemirror 的动态注入不可靠，与其在它内部和 reconfigure 时序搏斗，不如**绕过它**——lineWrapping 本质是给 `.cm-content` 加 class + CSS `white-space`，那直接用 CSS `!important` 强制。这比任何 compartment/reconfigure/重建方案都可靠。

**教训**：遇到封装层不可靠时，问自己「这个功能的本质是什么？最底层靠什么实现？」然后绕过封装层直接操作底层机制。CSS class 切换比编辑器实例重建/动态注入更稳定，且无副作用。

### 3. 诊断要基于运行时真实状态，而非静态分析

前两次修复都建立在「代码看起来应该生效」的静态分析上，但运行时并非如此。第三次加了运行时诊断（检查 `.cm-content` 的实际 class 和计算样式），才有可能确认方案是否真正生效。

**教训**：修复「不生效」类问题，**优先加运行时诊断**，把 DOM/状态的真实情况打出来，而不是反复改代码试错。基于真实状态的诊断，能避免在错误方向上越走越远。

### 4. 区分「初始化时序」和「运行时切换」两类场景

自动换行有两个生效场景：
- 启动时已开启（从持久化设置读取）——初始化时序问题。
- 运行时手动切换——watch/reconfigure 链路问题。

第一次修复只覆盖了初始化场景，漏了运行时切换。最终 CSS 兜底方案**同时覆盖两类场景**（因为 class 切换对两者一视同仁）。

**教训**：修「设置类」功能时，务必同时验证「启动时默认值」和「运行时切换」两个场景，避免修了一个漏了另一个。

### 5. CSS 兜底方案零副作用，优于实例重建

| | CSS 兜底 | :key 重建 |
|---|---|---|
| 光标位置 | ✅ 保留 | ❌ 丢失（需手动恢复） |
| 撤销历史 | ✅ 保留 | ❌ 丢失（无法恢复） |
| 选区 | ✅ 保留 | ❌ 丢失 |
| 滚动位置 | ✅ 保留 | ❌ 丢失（需手动恢复） |
| 可靠性 | 100%（纯 class 切换） | 依赖 vue-codemirror 机制 |

**教训**：能用 CSS 解决的视觉问题，优先用 CSS，不要动编辑器实例。实例重建是最后手段，且必然伴随状态丢失。
